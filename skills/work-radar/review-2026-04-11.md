# Skill Health Report: work-radar

Conversations analyzed: 9 (5 active, 4 correctly inactive/inconclusive)

## Overall Assessment

Work-radar's core value proposition — turning chat/meeting data into actionable operational intelligence — works when the question is a clean "today / status" query. The two daily check-in sessions and the extended todos session demonstrate the skill operating well: correct classification, honest confidence, fact/inference distinction, and usable output.

The skill breaks down in two specific situations:

1. **Thread-specific lookups** where the agent must navigate Slack's channel-ID-based file structure to find a specific person's DM (cursor 6f290e6c: 20+ tool calls just to locate the Seb DM, work-radar workflow completely bypassed)
2. **Multi-skill invocations** where work-radar is co-invoked with communication-copilot (6f290e6c and ses_28e2360caffe: the agent skips the work-radar output step and jumps straight to drafting)

The single largest efficiency problem across ALL sessions is **source discovery overhead** — the agent has to rediscover what's at each path, how Slack channels map to files, and where person-specific data lives. This costs 5-20+ tool call rounds per session that contribute zero operational value.

## Top Failure Modes (by frequency)

### 1. Navigation overhead / source rediscovery (5/5 active conversations)

- **Skill says:** "Likely source roots: /Users/stan/code/toolkit/brain/data/slack, /teams, /rec-*" (line 243-245)
- **What actually happened:** In every session, the agent spent multiple rounds listing directories, reading state.json and users.json to map channel IDs to names, and often hitting dead ends before finding the right files. In 6f290e6c, this consumed ~20 tool calls. In ses_29226109dffe, the agent's reasoning explicitly says "I need to gather data, likely from the rec data source... I may have to inspect the filesystem."
- **Root cause:** The skill lists directory paths but provides zero structural description of what's in them. Slack exports use opaque channel IDs (D0AM68RBJRG.json), requiring a state.json + users.json → channel name → file path lookup chain that the agent has to figure out every time. Critically, each workspace already has a `readable/` subdirectory with pre-rendered markdown files that bypass this entire problem — but the skill never mentions them.
- **Fix:** Add a source navigation guide as a reference file. Prominently direct agents to `readable/` subdirectories first.

### 2. Output template bypass (3/5 active conversations)

- **Skill says:** "Default output: Answer: / What looks active: / What may need follow-up: / Sources checked: / Confidence:" (lines 129-146)
- **What actually happened:** Only ses_29224bb22ffe and ses_29226109dffe used the exact template. ses_292ab2580ffe used a better variant. ses_28e2360caffe and 6f290e6c never produced any template output.
- **Root cause:** The template is positioned at the end of the workflow section. In longer conversations, the agent has moved past the instruction-reading phase by the time it needs to format output. Also, when the task naturally evolves beyond a simple lookup (interview prep, PR response), the template feels inapplicable and the agent drops it.
- **Fix:** Add a structural reminder before output generation. Make the template clearly adaptable for complex tasks.

### 3. Multi-skill boundary confusion (2/5 active conversations)

- **Skill says:** "switch to communication-copilot behavior only if explicitly asked" (line 232)
- **What actually happened:** In 6f290e6c, work-radar was invoked alongside communication-copilot, and the agent skipped work-radar's entire workflow, treating it as a data-location hint. In ses_28e2360caffe, the session drifted from work-radar territory into interview prep, scorecard filling, and reply drafting without a clear handoff.
- **Root cause:** The skill has no guidance for what happens when it's co-invoked with another skill. The interaction rules mention switching to communication-copilot but don't describe how to split the work.
- **Fix:** Add explicit multi-skill coordination guidance.

### 4. No execution-source cross-reference (1/5, but high-impact)

- **Skill says:** "Rank lower if it is: old and clearly resolved" (line 120)
- **What actually happened:** In ses_292ab2580ffe, the agent listed several tasks as open that were already merged PRs. The user had to prompt: "could you please check on GitHub the PRs... to check whether any of those items have been actually finished."
- **Root cause:** The skill only lists chat/meeting data as sources. It has no instruction to verify task status against execution sources (Git, PRs, CI).
- **Fix:** Add a verification step for "today / status" queries.
